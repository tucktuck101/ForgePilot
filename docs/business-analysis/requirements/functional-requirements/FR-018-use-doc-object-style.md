# FR-018: Use doc-object style

## Document Metadata

| Field | Value |
|---|---|
| Document type | Functional Requirement |
| Status | Draft |
| Owner/audience | Product Owner, Codex, future maintainers |
| Source | ForgePilot business analysis conversation |
| Linked artefacts | ../../index.md |
| Update rule | Update when this functional requirement changes materially |

## Summary

All major documents generated by ForgePilot (briefs, requirements, personas, user stories, and design artefacts) shall follow a standard doc-object style to ensure clarity and consistency.

## Details

The doc-object style includes:

- A top-level heading with the ID or name and title.
- A Document Metadata table specifying the document type, status, audience, source, linked artefacts, and update rule.
- A Summary section that provides a short overview.
- A Details section containing the body of the document.
- An Acceptance Criteria section when applicable (e.g., requirements and user stories).
- Optional Notes, assumptions, or traceability information.

Using this structure makes artefacts self-describing and easier to maintain.

## Acceptance Criteria

- All discovery documents, requirements, persona files, and user stories follow the doc-object structure.
- Metadata fields are completed appropriately for each document.
- The document type field reflects the type (e.g., Business Requirement, Persona, User Story).

## Notes

Adhering to a consistent documentation style is crucial for both human and AI agents to quickly understand the purpose and status of each artefact.
